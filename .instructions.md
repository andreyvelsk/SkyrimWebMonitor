# SkyrimWebMonitor - System Instructions for Copilot Agents

## Project Overview
**SkyrimWebMonitor** — Vue 3 PWA приложение для мониторинга игрового состояния The Elder Scrolls: Skyrim. Real-time WebSocket мониторинг, многоязычная поддержка, адаптивный дизайн для мобильных устройств.

**Tech Stack:**
- Vue 3 (Composition API)
- TypeScript (strict mode enabled)
- Vite + PWA support
- Pinia (state management)
- vue-i18n (en, ru locales)
- Tailwind CSS + Sass/SCSS
- ESLint + Prettier (auto-fix enabled)

## Mandatory Architecture Principles

### 1. Feature-Sliced Design (FSD) Compliance
```
src/
├── app/              # Application core (routing, config, root UI)
│   ├── config/       # Page registry, feature configs
│   ├── router/       # Navigation logic
│   └── ui/           # Root layout components
├── pages/            # Feature modules (character, inventory, magic, map)
│   ├── [feature]/
│   │   ├── index.ts  # Export barrel file
│   │   ├── *.vue     # Component files
│   │   ├── types/    # Feature-specific types
│   │   └── composables/
├── stores/           # Pinia stores (by domain)
│   ├── [domain]/
│   │   ├── use[Domain]Store.ts
│   │   ├── types.ts
│   │   └── adapters/
├── shared/           # Cross-cutting concerns
│   ├── lib/          # Utilities, composables
│   │   ├── composables/
│   │   ├── config/
│   │   ├── constants/
│   │   ├── types/
│   │   └── styles/
│   └── ui/           # Shared components
├── i18n/             # Internationalization
│   ├── index.ts
│   └── locales/      # en.json, ru.json
└── api/              # External integration
    └── websocket/    # Protocol implementation
```

**FSD Rules:**
- ✅ Slices are independent; import from shared only
- ✅ Circular dependencies are prohibited
- ✅ Types live in separate `types/` directories
- ✅ Export barrel files (`index.ts`) from each public API level
- ✅ Components start with capital letters: `TheCharacter.vue`, `InventoryItem.vue`

### 2. Type Organization (STRICT)
**All types must be in separate files in `types/` directories:**

```typescript
// ✅ CORRECT: src/pages/character/types/types.ts
export interface CharacterStats {
  health: number;
  magicka: number;
}

// ✅ CORRECT: src/stores/character/types.ts
export type CharacterStoreState = {
  stats: CharacterStats;
};

// ❌ WRONG: Inline type definitions in .vue or .ts files
// ❌ WRONG: No dedicated types/ directory
```

**Type imports must use full paths:**
```typescript
import type { CharacterStats } from '@/pages/character/types/types';
import type { CharacterStoreState } from '@/stores/character/types';
```

### 3. Internationalization (i18n) - MANDATORY
**Every UI string must use i18n:**

```typescript
// ✅ CORRECT: Use t() function
<template>
  <button>{{ t('buttons.save') }}</button>
</template>

<script setup lang="ts">
import { useI18n } from 'vue-i18n';
const { t } = useI18n();
</script>

// ❌ WRONG: Hardcoded strings
<button>Save</button>

// ❌ WRONG: Concatenated i18n keys
{{ t('prefix') + 'suffix' }}
```

**Locale files structure:** `src/i18n/locales/{en,ru}.json`
- Always add keys to BOTH locale files
- Use dot notation for nested keys: `character.stats.health`
- Group by feature: `character.*`, `inventory.*`, `map.*`

### 4. Vue 3 Best Practices

#### Composition API Only
```typescript
// ✅ CORRECT
<script setup lang="ts">
import type { Ref } from 'vue';
import { ref, computed, watch } from 'vue';
import { Props, Emits } from './types';

withDefaults(defineProps<Props>(), {
  modelValue: '',
});
const emit = defineEmits<Emits>();
</script>

// ❌ WRONG: Options API or legacy patterns
```

#### Component Naming & Organization
```typescript
// ✅ Component names use PascalCase with semantic names
TheCharacter.vue      // Feature component
InventoryItem.vue     // Presentational component
SkyrimNavigation.vue  // Layout component

// ❌ WRONG
CharacterPage.vue (should be TheCharacter.vue)
Item.vue (should be InventoryItem.vue)
```

#### Proper Props/Emits Declarations
```typescript
// ✅ CORRECT
interface Props {
  id: string;
  label: string;
  disabled?: boolean;
}
interface Emits {
  (e: 'click', id: string): void;
  (e: 'hover:enter'): void;
}

withDefaults(defineProps<Props>(), {
  disabled: false,
});
const emit = defineEmits<Emits>();

// ❌ WRONG
props: {
  id: String,
  label: String,
}
```

#### Reactive State Management
```typescript
// ✅ CORRECT: Use ref for primitive types, reactive() only if needed
const count = ref(0);
const items = ref<Item[]>([]);

// ✅ CORRECT: Use computed for derived state
const isLoading = computed(() => store.loading);

// ❌ WRONG: Direct DOM manipulation
document.querySelector('.item').textContent = 'test';

// ❌ WRONG: Watchers for side effects only
// Use immediate computed instead when possible
```

#### Setup Patterns
```typescript
// ✅ CORRECT: Initialize in root scope, use watchers for side effects
const { t } = useI18n();
const store = useNavigationStore();
const { activeTab } = storeToRefs(store);

onMounted(() => {
  store.loadInitialData();
});

// ❌ WRONG: Complex logic in onMounted
onMounted(() => {
  if (props.id) {
    store.fetch(props.id);
    watch(activeTab, () => {
      // ...
    });
  }
});
```

### 5. Pinia Stores Structure

**Store organization by domain:**
```typescript
// ✅ CORRECT: src/stores/character/useCharacterStore.ts
export const useCharacterStore = defineStore('character', () => {
  // State
  const stats = ref<CharacterStats>({...});
  
  // Getters (computed)
  const isHealthLow = computed(() => stats.value.health < 30);
  
  // Actions
  const updateStats = (newStats: CharacterStats) => {
    stats.value = newStats;
  };
  
  return {
    // State
    stats,
    // Getters
    isHealthLow,
    // Actions
    updateStats,
  };
});

// Always use: storeToRefs() for reactive refs
<script setup lang="ts">
const store = useCharacterStore();
const { stats, isHealthLow } = storeToRefs(store);
const { updateStats } = store;
</script>
```

**Store adapters for complex data transformations:**
```
src/stores/adapters/
├── typeGuards.ts      # Type safeguards
├── dataRouter.ts      # Route incoming data to correct stores
└── localeAdapter.ts   # Localization mapping
```

## Development Workflow

### When Creating New Features

1. **Create feature directory** in appropriate slice:
   ```bash
   src/pages/quests/
   ├── index.ts           # Export barrel
   ├── TheQuests.vue      # Main component
   ├── types/
   │   └── types.ts       # Feature types
   └── composables/
       └── useQuestData.ts
   ```

2. **Add types first:**
   ```typescript
   // src/pages/quests/types/types.ts
   export interface Quest {
     id: string;
     name: string;
     status: 'active' | 'completed';
   }
   ```

3. **Create component with i18n:**
   ```typescript
   // src/pages/quests/TheQuests.vue
   <template>
     <section class="quests">
       <h2>{{ t('quests.title') }}</h2>
     </section>
   </template>
   ```

4. **Add i18n keys to both locales:**
   ```json
   // src/i18n/locales/en.json
   "quests": {
     "title": "Quests"
   }
   
   // src/i18n/locales/ru.json
   "quests": {
     "title": "Квесты"
   }
   ```

5. **Export from barrel:**
   ```typescript
   // src/pages/quests/index.ts
   export { default as TheQuests } from './TheQuests.vue';
   export type * from './types/types';
   ```

### When Modifying Existing Code
1. Maintain FSD structure — don't relocate types
2. Update both locale files if UI strings change
3. Ensure all types exported from `types.ts`
4. Run `npm run lint` + `npm run format` before completion
5. Check TypeScript with `npm run tsc`

### WebSocket Integration
- Message validation happens in `src/api/websocket/protocol.ts`
- Store subscriptions managed in `useWebSocketStore`
- Dynamic page subscriptions via `CategorySubscriptionConfig`

## Import Aliases & Module Resolution

**Always use `@/*` alias for src imports:**
```typescript
// ✅ CORRECT
import type { CharacterStats } from '@/pages/character/types/types';
import { TheCharacter } from '@/pages';
import { useCharacterStore } from '@/stores/character/useCharacterStore';

// ❌ WRONG
import type { CharacterStats } from '../../../pages/character/types/types';
import { TheCharacter } from '@/pages/character';
```

**Barrel files (`index.ts`) principle:**
- Always export from public API in slice
- Never import from private implementation details
- Keep dependencies explicit and clear

## Common Patterns & Anti-Patterns

### ✅ DO
- Use `storeToRefs()` for reactive store properties in templates
- Separate presentation logic from business logic
- Use composables for reusable logic across components
- Type function parameters and return types explicitly
- Use enums or literals for status values: `'active' | 'completed'`
- Keep components small and focused

### ❌ DON'T
- Import deeply nested components directly
- Put i18n keys in JavaScript strings
- Mix Composition API with Options API
- Create types outside of `types/` directories
- Use default exports for stores or utilities
- Commit without running linters
- Create circular dependencies between slices

## Debugging Tips
1. **i18n key not found?** Check both `en.json` and `ru.json` have the key
2. **Type errors in stores?** Verify `types.ts` is exported from barrel
3. **Component not rendering?** Check PascalCase naming + barrel export
4. **WebSocket issues?** Validate message types in `protocol.ts`
